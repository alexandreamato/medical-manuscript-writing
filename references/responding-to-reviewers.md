# Responding to Reviewers (Medical Manuscripts)

This file covers the post-submission stage: how to read a decision letter, write a point-by-point response, prepare a tracked-changes manuscript, and resubmit. The response letter is among the highest-leverage documents in the publication process — a strong response can convert a "major revision" into an acceptance, and a weak one can turn a fixable manuscript into a rejection.

For pre-submission work, see `references/writing-process.md` and `references/paper-review.md`.

## When You Receive the Decision

Most decision letters fall into one of five categories:

| Decision | Typical interpretation | Response strategy |
| --- | --- | --- |
| Accept (rare on first round) | The paper is publishable as is. | Address minor formatting / typesetting; thank the editor. |
| Minor revision | Generally favourable; reviewers want clarifications, not new analyses. | Make every requested change; concise responses; turnaround in 2–4 weeks. |
| Major revision | Editor sees promise but reviewers raised substantive issues. | Most common path to publication. Treat seriously; turnaround 4–8 weeks. |
| Reject and resubmit | Editor offers a path back if the issues are addressed. | Same approach as major revision; resubmission is a new submission. |
| Reject | Paper is not a fit, or the issues are unfixable for this journal. | Read carefully; revise based on the feedback; submit elsewhere. |

A "major revision" decision is not a rejection. The historical acceptance rate after major revision is high in most journals (often 60–80%) **if the response is thorough and respectful**.

## Read the Decision Letter Twice, Calmly

The first read is emotional. The second is analytical. Wait at least 24 hours between them.

On the second read, classify each reviewer comment as one of:

1. **Factual error or typo to fix** — quick, no debate.
2. **Reasonable request for clarification** — add or rephrase text.
3. **Reasonable request for new analysis** — do it if feasible; explain if not.
4. **Question whose answer is already in the manuscript** — the reviewer missed it; respond by pointing to the existing text **and improve the visibility** (clearer wording, better signposting).
5. **Reviewer factual error** — disagree, but respectfully and with evidence.
6. **Reviewer scope creep** — the request goes beyond the paper's aim. Respond by reaffirming scope; offer the request as a future direction.
7. **Conflict between reviewers** — the editor decides; flag the conflict explicitly in your response.

Do not classify any comment as "the reviewer is wrong, ignore". Even unreasonable requests need an answer.

## Read All Comments Before Drafting Anything

Before opening a Word document or starting to edit the manuscript, **read every comment from every reviewer end-to-end** at least once. Reviewers often disagree, and editing in response to one comment that another reviewer wants deleted is wasted work. Common patterns to detect on the first read:

1. Reviewer 1 asks you to expand a paragraph; Reviewer 2 says the same paragraph should be deleted.
2. Reviewer 1 asks for a new analysis that, if done, would change Table 2 — and Reviewer 2 has comments on Table 2 that assume the current version.
3. Two reviewers ask the same question with different wording — answer once and link.

Map the conflicts before starting. See "Handling Conflicting Reviewer Comments" below.

## Document Structure for Resubmission

Most journals expect three documents at resubmission:

1. **Cover letter to the editor** — short; thank the editor and reviewers; summarize the main changes; flag any sensitive issues.
2. **Point-by-point response letter** — your detailed reply to every reviewer comment.
3. **Revised manuscript** in two versions:
   - **Tracked-changes / highlighted version** showing every edit.
   - **Clean version** with all changes accepted and ready for production.

Some journals also require an updated reporting checklist (CONSORT, STROBE, PRISMA, STARD, CARE). Update it.

## Three Formats for the Response Document

Three layouts are common; pick one before starting and stick with it.

### Format 1 — Discursive (recommended)

A linear document with reviewer comments in **bold**, your response in plain text below each, and a blank line between blocks. Most editors and reviewers find this the easiest to read.

```
Response to Reviewers
"[Manuscript title]"
[Author names]

The reviewer comments are reproduced verbatim in bold below. Author replies
follow each comment in plain text. Changes are highlighted in yellow in the
revised manuscript; line numbers refer to the revised manuscript.

Reviewer #1

R1.1   [Reviewer comment in bold, copied verbatim from the decision letter.]

       Response: [Author reply in plain text.]

       Manuscript change (Methods, page 7, lines 142–148):
       "[Quoted new text.]"

R1.2   ...

Reviewer #2

R2.1   ...

Other changes
[List any edits not driven by reviewer comments — e.g., updated reference,
fixed typo on page 4 line 88, regenerated Figure 2 with corrected colour
palette.]
```

### Format 2 — Three-column table

Some journals prefer or require a tabular response: column 1 reviewer comment, column 2 author response, column 3 manuscript location of the change. Cleaner for very long lists; harder to read for nuanced disagreement.

### Format 3 — Inline in the journal portal

Some online submission systems offer a free-text "Response to Authors" field. Generally less convenient than uploading a separate document; use only when the journal explicitly requests it. If you go inline, still keep a complete response document on file for your records.

## Always Repeat Reviewer Comments Verbatim

Copy and paste each comment exactly from the decision letter — no editing, no rephrasing, no spelling or grammar fixes. Reviewers often skim only the comments they wrote; if you paraphrase or summarize, they cannot find their own words and conclude you did not address their point.

If a reviewer's comment is convoluted (multiple sub-points jumbled together), you have two options:

1. **Break it into sub-points** (R1.4a, R1.4b, R1.4c) and respond to each.
2. **Leave it as a single block** if breaking it apart would distort meaning, then provide a single overarching response.

Pick the more honest option, not the easier one.

## Cover Letter at Resubmission

One page. Structure:

```
Dear Dr. [Editor name],

Thank you for the opportunity to revise our manuscript "[Title]" (Manuscript ID: [ID]).
We are grateful to you and to the reviewers for the careful and constructive comments,
which have substantially strengthened the paper.

We have addressed every comment in the accompanying point-by-point response. The most
important changes include:

  - [Change 1, one sentence]
  - [Change 2, one sentence]
  - [Change 3, one sentence]

We have provided a tracked-changes manuscript and a clean version. The completed
[CONSORT / STROBE / PRISMA / STARD / CARE] checklist is updated and attached.

We confirm that the manuscript is not under consideration elsewhere and that all
authors have approved the revised version.

We hope the revised manuscript meets your expectations and look forward to your
decision.

Sincerely,

[Corresponding author]
```

## Point-by-Point Response — Format

The response document is the heart of the resubmission. Most journals expect a specific format. Build it as a single document with reviewer comments in one style and your responses in another, so the editor can scan it.

### Recommended formatting

1. **Header per reviewer** in bold: `Reviewer 1`, `Reviewer 2`, `Editor`.
2. **Reviewer comment** quoted verbatim, in italics or in a box.
3. **Author response** below, in regular text, prefixed with `Response:`.
4. **Manuscript change** with the exact new text quoted in quotation marks and the page or line number.
5. **Numbered every comment** continuously per reviewer (`R1.1`, `R1.2`, `R2.1`, ...).

### Worked example

```
Reviewer 1

Comment R1.1
"The sample size calculation in the Methods is unclear. What effect size was assumed
and on what basis?"

Response:
We thank the reviewer for this important point. The sample size calculation was based
on a 5-percentage-point absolute risk reduction (24% to 19%) observed in the pilot
study by Smith et al. (Reference 12). This assumption has now been stated explicitly
in the Methods.

Manuscript change (Methods, page 7, lines 142–148):
"Assuming a primary outcome event rate of 24% in the control arm and an absolute risk
reduction of 5 percentage points (relative risk 0.79), based on the pilot data
reported by Smith et al. [12], a two-sided alpha of 0.05, and 80% power, the trial
required 1,150 participants per arm. We inflated this to 1,250 to allow for 8% loss
to follow-up."

Comment R1.2
"Why was the per-protocol analysis chosen as the primary analysis rather than ITT?"

Response:
We respectfully clarify that the prespecified primary analysis is in fact the
intention-to-treat analysis (Methods, page 8, lines 168–170; results in Table 2 and
Figure 2). The per-protocol analysis is reported as a sensitivity analysis only
(Table S2). We have rephrased the Methods to make this distinction clearer.

Manuscript change (Methods, page 8, lines 168–172):
"The primary analysis followed the intention-to-treat principle and included all
randomized participants (n = 2,450). A per-protocol analysis (n = 2,210), restricted
to participants who received the assigned intervention without major protocol
deviations, is reported as a sensitivity analysis (Supplementary Table S2)."
```

## The 70% Rule (Efron 2025)

A practical heuristic: **respond positively to at least 70% of reviewer comments**. If your response document is mostly arguments for *not* changing things, the reviewer feels shunned and rejects.

This means picking your fights carefully. Most comments deserve a substantive change to the manuscript; reserve "we respectfully retain" for comments where the design or evidence genuinely cannot support the requested change. When in doubt, accommodate. Many comments that feel unreasonable on first read can be addressed with a small clarification or limitation paragraph.

## Three Adages for the Right Stance (Efron 2025)

When framing how you respond — especially to critical or borderline-aggressive comments — keep three classical sayings in mind:

1. **"Rise above the fray."** Do not get into acrimonious back-and-forth with the reviewer. Lift the response to the level of common practice or established convention.
2. **"Take the high ground."** Frame your approach as gold-standard practice, aligned with a leading authority. Cite the reference: "We followed the approach of [Author Year], which has become standard for ..."
3. **"Play with a straight bat."** A cricket analogy: play orthodox, stay focused on the question at hand, do not stray into spurious arguments or extraneous discussions. Adhere to scientific orthodoxy and let the orthodoxy carry the response.

## Never Assume the Reviewer Is Stupid (Cushman 2023)

Reviewers and editors may have less topic-specific expertise than you. If your gut reaction is "this person does not understand", step back and consider:

1. **Language barrier.** English is not everyone's first language; the comment may be technically correct but awkwardly phrased. Re-read assuming good intent.
2. **Different perspective, not lower expertise.** A statistician reviewer may flag something a clinician reviewer would have ignored, and vice versa. Each lens is legitimate.
3. **A genuinely off-topic comment** is rare; usually the reviewer is responding to something real you did not foreground enough.

If after careful reading the comment is still unclear, **say so politely** rather than guessing:

```
Response: We are not certain we have understood the reviewer's question correctly.
We interpret it as asking [our interpretation], and we have responded by
[action]. If we have missed the intended meaning, we welcome the opportunity
to address it again.
```

## Tone — Constructive, Specific, Never Defensive

Reviewers are giving free time to improve a paper. Even when a comment seems unfair, the response sets the editor's impression of the authors. Five rules:

1. **Thank the reviewer** at least once at the start (not for every comment).
2. **Answer in full, not in part.** Skipping or partially addressing comments invites a second round.
3. **Quote the changed text** with page or line number. Do not just say "we have changed the manuscript".
4. **Disagree with evidence, not with adjectives.** Cite a paper, point to the data, or explain the methodological reason.
5. **Avoid defensive vocabulary.** Words like "obviously", "clearly", "the reviewer misunderstood", "as we already wrote" are taken badly.

### Phrases that work

```
We thank the reviewer for raising this important point.
We agree with the reviewer and have ...
We have clarified this in the revised manuscript ...
We respectfully disagree, for the following reason: ...
We acknowledge the limitation and have added a sentence to the Discussion ...
This is an important suggestion. While we cannot perform the additional analysis
because [specific reason], we have added a paragraph to the Discussion explaining ...
```

### Phrases that backfire

```
"As we already wrote ..." (sounds dismissive even when true)
"The reviewer is mistaken." (better: "We may not have explained this clearly")
"This is outside the scope of the paper." (better: "This is an important question that
we agree warrants future study, but it falls outside the prespecified aims of the
present analysis.")
"We have added a sentence." (vague — quote it)
```

## Disagreeing Respectfully

Sometimes a reviewer comment is wrong, unreasonable, or asks for something the design cannot deliver. Disagreement is legitimate; how you do it matters.

### Pattern

1. Acknowledge the reviewer's concern.
2. State the disagreement explicitly.
3. Provide the evidence for your position (citation, data, prespecified plan, methodological argument).
4. Offer a reasonable accommodation (e.g., add a sentence to the Discussion, add a sensitivity analysis even if you do not change the primary).

### Example

```
Comment R2.5
"The authors should re-analyze the data using a Cox model rather than logistic
regression."

Response:
We thank the reviewer for this suggestion. We respectfully retain the logistic-
regression approach because the primary outcome was measured at a single, fixed
time point (12 months after randomization), and not all participants were followed
to a later event. A Cox model assumes time-to-event data with right censoring, which
does not apply to a binary outcome at a fixed timepoint (Sterne et al., BMJ 2005).
We acknowledge that a Cox model would be appropriate if the primary outcome were
time to first event; we have added a sentence to the Methods clarifying this design
choice and a sentence to the Limitations noting that a time-to-event analysis was
not feasible because we did not collect event-date data.

Manuscript change (Methods, page 9, lines 187–189):
"Logistic regression was selected because the primary outcome was assessed at a
fixed timepoint (12 months) rather than as time to event."

Manuscript change (Discussion, page 22, lines 478–481):
"This study did not collect event dates for the primary outcome, precluding a
time-to-event analysis. Future trials with continuous follow-up should consider a
Cox proportional-hazards approach."
```

## The "Sham Response" for Vague Reviewer Suggestions (Efron 2025)

Sometimes a reviewer makes a vague suggestion such as "the first paragraph is unclear and should be redrafted" — without explaining *why* it is unclear — even when you are confident the paragraph is fine. Defending the existing paragraph with "We are confident that the paragraph is clear as written" usually fails because the reviewer feels ignored.

The pragmatic move: **rewrite the paragraph using different words and tell the reviewer it has been redrafted for clarity**. The new version may not be objectively clearer, but the reviewer sees evidence of effort. Efron calls this the "sham response" — not a real fix, but a tactical accommodation that keeps the relationship constructive.

```
Response: We thank the reviewer for this suggestion. The opening paragraph has been
redrafted to enhance clarity (Introduction, page 2, lines 28–35).

Manuscript change (Introduction, page 2, lines 28–35):
"[The rewritten paragraph, using different wording from the original.]"
```

Use sparingly — only when the original was already clear and the reviewer's complaint was unspecific. Do not use this technique to dodge substantive criticism.

## Common Reviewer Criticisms — Defensible Countermoves

Two complaints recur in almost every set of reviewer comments. Have prepared responses ready.

### Criticism 1 — "The sample size is too small"

This complaint is only relevant if your paper reports a **non-significant** finding. If you report a statistically significant result, the sample size **must** have been sufficient — point this out directly:

```
Response: We thank the reviewer for raising this point. The primary outcome reached
statistical significance (HR 0.78; 95% CI 0.66 to 0.92; p = 0.003), which
indicates that the sample size was sufficient to detect the effect of interest.
The sample size was prespecified at 1,250 per arm based on [assumptions; cite
protocol]; we have added the assumed effect size and source explicitly to
Methods (page 7, lines 142–148).
```

If you report a **non-significant** finding and the reviewer is concerned about underpowering, perform a post-hoc power analysis to demonstrate adequate power and report the result. Do **not** dismiss the concern.

### Criticism 2 — "The Discussion strays beyond the reported work"

This is often a misunderstanding of what a Discussion should do. The Discussion section's role *is* to extend thinking and demonstrate how the work advances the field. Argue back firmly:

```
Response: We respectfully retain this aspect of the Discussion. An important function
of the Discussion section is to interpret the present findings in the broader
context of the literature and to indicate the directions in which the work
extends current knowledge (citing Hess 2023 on effective Discussion writing,
Mateu Arrom et al. 2018, and the ICMJE Recommendations). We have, however,
re-read the section and tightened two paragraphs (page 22, lines 462–478) to
ensure each speculative comment is explicitly flagged with phrases such as "We
speculate that ..." and "Future work will be needed to determine whether ...".
```

### Criticism 3 — "Add P values to Table 1" (Cushman 2023)

A reviewer asks for hypothesis-test P values comparing baseline characteristics across groups. This is methodologically incorrect for randomized trials (CONSORT discourages it) and unhelpful for observational studies (multiple comparisons, distracting from confounder identification). Defensible response:

```
Response: We respectfully prefer not to add P values to Table 1, for four reasons:
(1) Table 1 is descriptive, not hypothesis-testing; baseline characteristics
should be summarized, not compared inferentially. (2) Statistical inference is
not a valid method for identifying confounders; standardized mean differences
are preferred. (3) Inferential measures in baseline tables address whether
exposure and a covariate are associated in the underlying population, which
is tangential to the study hypotheses. (4) Multiple comparisons in baseline
tables generate false-positive associations. CONSORT 2010 also discourages
this practice in randomized trials. We hope the reviewer and editor agree.
However, we can add the P values at the discretion of the Editor.
```

The closing line ("at the discretion of the Editor") is a soft escape valve when you anticipate editor pushback.

## Handling Conflicting Reviewer Comments

When two reviewers ask for opposite changes, you cannot satisfy both. Flag the conflict to the editor explicitly:

```
Comment R1.7
"The methods section is too long; please condense."

Comment R3.4
"The methods section needs more detail on the statistical analysis plan."

Response:
We note that R1.7 asks us to shorten the Methods, while R3.4 asks for more detail.
We have addressed R3.4 by adding a brief paragraph (Methods, page 9, lines 187–198)
on the statistical analysis plan, because the additional detail aids reproducibility.
We have offset the addition by tightening the description of the recruitment process
(Methods, page 6, lines 118–125), which we agree with R1.7 was overly long.
```

## Handling Comments You Cannot Address

Some requests cannot be met:

1. **Resource constraints** — additional data collection, additional clinical samples.
2. **Design constraints** — the trial is finished; randomization cannot be re-done.
3. **Ethics constraints** — IRB approval did not cover the proposed additional analysis.
4. **Statistical constraints** — sample size insufficient for a stratified analysis the reviewer wants.

Pattern: acknowledge, explain why, offer the best feasible substitute.

```
Response:
We agree that an external validation cohort would strengthen the paper. We do not
have access to such a cohort within the timeline of this resubmission and the data
sharing agreements with our partner sites. We have therefore added a paragraph to
the Discussion (page 23, lines 510–518) explicitly identifying external validation
as the most important next step, and we have shared the model code and a synthetic
example dataset (Supplementary File 4) so that other groups can perform external
validation independently.
```

## Repeated Comments — Duplicate, Don't Cross-Reference

A reviewer sometimes raises the same concern twice in different parts of their review (perhaps with different wording). Two reviewers may also raise the same concern independently.

Do **not** write "see response to comment R1.3" and leave it at that. Reviewers may only read responses to *their own* comments, and a cross-reference forces them to scroll to find an answer that should be in front of them. Instead, **duplicate the entire response** with a soft link:

```
R1.7  [Comment about the missing-data approach.]
      Response: As explained previously (R1.3), missing data on the primary
      outcome were handled by multiple imputation under MAR, with [details].
      Manuscript change ...
```

Same rule for comments raised by two different reviewers. Repeat the response in full under each reviewer.

## Respond to Author Instructions, Not Just Reviewer Comments (Cushman 2023)

Decision letters often contain editorial requests that are not reviewer comments — formatting requirements, search-engine-optimization steps, race/ethnicity reporting, data-sharing statements, ORCID IDs, conflict-of-interest forms. **Address every one** under a separate heading in the response document:

```
Editorial requests

E.1   The journal asks that all manuscripts be optimized for search engines.
      Response: We have revised the Title to include the key terms [...] and
      added [...] to the Abstract and keywords.

E.2   The journal asks for race and ethnicity data on participants.
      Response: We have added a sentence to Methods (page 7, lines 158–160)
      describing the racial and ethnic composition of the sample. We have
      also added a sentence to Limitations noting that all participants were
      recruited from a single tertiary centre in [city], which limits
      generalizability across populations.
```

Skipping editorial instructions is a common cause of desk rejection at resubmission. Editors notice.

## Word Limits — Strategies When You Need More Space

The revised manuscript must usually still fit the journal's word limit, even after substantive additions. Strategies in order of preference:

1. **Tighten existing prose first.** Most manuscripts can absorb a 10–15% reduction without losing content; see `references/scientific-writing-principles.md` for the wordy → concise table.
2. **Move added material to Supplementary.** Sensitivity analyses, full search strategies, additional figures, expanded Methods detail can all live in the supplement, which is not counted.
3. **Drop content the reviewer indirectly authorized you to drop.** If a reviewer asked you to expand X, the editor will usually tolerate a small contraction of Y.
4. **Ask the editor for leeway** with a brief note in the cover letter: `The reviewers' substantive requests have required adding approximately 280 words to the Methods. We hope the editor will allow a small extension above the 3,500-word limit.` Editors usually grant reasonable extensions.

## Tracked-Changes Manuscript

Submission of a marked-up version is mandatory at most journals. Practical rules:

1. **Use Word's Track Changes** with all author edits visible. Do not accept changes before submission of the tracked version.
2. **Highlight new text** in a colour (yellow is conventional) when Track Changes will not show large block additions clearly.
3. **Preserve the manuscript line numbering** so the reviewer can match your response to the location.
4. **Submit the clean version** with all changes accepted, line numbers preserved, ready for production.
5. **Do not delete reviewer-quoted text** in the response — it is the anchor of the conversation.

## Timing

| Decision | Typical turnaround |
| --- | --- |
| Minor revision | 2–4 weeks |
| Major revision | 4–8 weeks |
| Reject and resubmit | 6–12 weeks (treat as a fresh submission) |

Do not rush. A response that arrives in 5 days but is incomplete looks worse than one that arrives in 5 weeks and is thorough. If you need more time, write to the editor — most journals grant extensions.

## After Resubmission — Possible Outcomes

1. **Accept (with or without further minor changes).** Address final formatting; respond to any final editor queries; check the proofs carefully (see `references/writing-process.md`, "After Acceptance").
2. **Second round of revision.** Treat the same way: point-by-point response, tracked changes. Some journals limit the number of revision rounds; verify.
3. **Reject after revision.** Read the editor's letter carefully. If the rejection is for fit rather than fundamental flaws, consider another journal. The point-by-point response and tracked-changes file you prepared are reusable.

## Quality Checklist Before Sending the Resubmission

1. Every reviewer comment has a numbered response.
2. Every response quotes the new text and gives the page or line number.
3. Tone is constructive throughout (re-read after 24 hours).
4. Disagreements are evidence-based, not rhetorical.
5. Tracked-changes and clean versions both attached.
6. Cover letter summarizes the 3 to 5 most important changes.
7. Updated reporting checklist (CONSORT / STROBE / PRISMA / STARD / CARE) is included if required.
8. References, tables, and figures renumbered if positions changed.
9. Author list, ORCIDs, affiliations, and acknowledgments verified.
10. Manuscript is internally consistent: numbers in the abstract still match the Results; the Discussion still matches the new analyses.
